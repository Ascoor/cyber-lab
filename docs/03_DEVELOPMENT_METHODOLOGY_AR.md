# منهجية التطوير الإلزامية

## قبل التنفيذ
- اقرأ `docs/01_ROADMAP_AR.md` وحدد المرحلة الحالية.
- اقرأ `docs/02_SECURITY_AND_SCOPE_POLICY_AR.md` وتأكد أن التعديل دفاعي ومحدود.
- حدد بوضوح الملفات التي تحتاج تعديلًا.
- لا تضف ميزة خارج المرحلة الحالية.
- لا تبدأ تنفيذ موديول جديد قبل تعبئة قالب `docs/08_MODULE_SPECIFICATION_TEMPLATE_AR.md` أو إنشاء مواصفة مبنية عليه.

## أثناء التنفيذ
- عدل أقل عدد ممكن من الملفات.
- اجعل الكود واضحًا ومقسمًا إلى دوال صغيرة عند الحاجة.
- لا تستخدم `shell=True`.
- لا تسمح بإدخال أوامر أو flags مباشرة من المستخدم.
- أضف validation واضح للمدخلات، خصوصًا target وauthorized ونوع الفحص.
- أضف timeout لأي عملية خارجية.
- تعامل مع الأخطاء برسائل واضحة دون كشف أسرار.
- لا تضف dependencies كبيرة إلا عند وجود قرار معماري موثق.
- لا تحذف endpoints أو وظائف موجودة إلا بقرار موثق وسبب واضح.

## بعد التنفيذ
- شغل اختبار سريع مناسب للتعديل.
- شغل `python -m compileall backend/app` عند تعديل Backend Python.
- اختبر `/health` إذا كان backend قابلًا للتشغيل.
- اختبر الحالة المسموحة والحالة المرفوضة عند تعديل أي فحص.
- راجع أثر التعديل على النطاق القانوني والأمني.
- حدث `docs/05_CHANGELOG_AR.md`.
- حدث `docs/06_DECISION_LOG_AR.md` إذا وُجد قرار معماري أو أمني جديد.
- حدث `README.md` إذا تغيرت طريقة الاستخدام أو التشغيل.
- اكتب ملخص تنفيذ مختصر يوضح ما تغير وما لم يتغير.

## قواعد مراجعة الكود
- أي كود يشغل أداة خارجية يجب مراجعته ضد command injection.
- أي endpoint جديد يجب توثيقه مع أمثلة اختبار.
- أي تقرير جديد يجب أن يلتزم بمعيار `docs/09_REPORTING_STANDARD_AR.md`.
- أي فحص جديد يجب أن يرفض `authorized=false` قبل أي تنفيذ.
